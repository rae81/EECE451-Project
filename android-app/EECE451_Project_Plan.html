Project Overview

What You're Building

An Android application named "Network Cell Analyzer" that collects real cellular network data from the serving base station your phone is connected to. It supports GSM/GPRS/EDGE (2G), UMTS (3G), and LTE (4G) networks.

Component A: Android App

  • Reads real cell tower data from phone radio
  • Displays real-time: Operator, Signal Power, SINR/SNR, Network Type, Frequency Band, Cell ID
  • Sends data to server every ~10 seconds
  • Fetches and displays historical statistics with date range filters
  • Supports 2G, 3G, and 4G networks

Component B: Backend Server

  • Accepts data from multiple phones simultaneously
  • Stores all cell data in a database
  • Computes statistics (avg signal, connectivity %, SNR, etc.)
  • Serves statistics back to the app on request
  • Has a web dashboard showing connected devices (IP, MAC)

API Endpoints Design

POST   /receive-data

Receive cell data from a phone. Called every ~10 seconds by the Android app.

// Request Body (JSON) { "device_id": "abc123-def456", "operator": "Alfa", "signal_power": -85, "snr": 12.5, "network_type": "4G", "frequency_band": "Band 3 (1800MHz)", "cell_id": "37100-81937409", "timestamp": "09 Mar 2026 02:30 PM" } // Response: 201 { "message": "Data received" }

GET   /get-stats?device_id=...&start=...&end=...

Fetch per-device statistics between two dates.

// Response: 200 { "connectivity_per_operator": { "Alfa": "75.5%", "Touch": "24.5%" }, "connectivity_per_network_type": { "4G": "70%", "3G": "25%", "2G": "5%" }, "avg_signal_per_network_type": { "4G": -82.3, "3G": -91.5, "2G": -98.1 }, "avg_snr_per_network_type": { "4G": 15.2 }, "avg_signal_device": -85.7 }

GET   /central-stats

Server web dashboard (HTML page). Shows connected devices, IPs, MACs, and status.

GET   /get-stats/avg-all?start=...&end=...

Cross-device averages (signal and SNR across all phones).

Innovative Extra-Credit Features

Strategy for Maximum Extra Points: Implement at least 4-5 of these features to push your grade well above 100%. Each feature is designed to demonstrate advanced skills that go beyond the base requirements. The estimated point values are based on complexity and impressiveness.
1. Real-Time Signal Heatmap with GPS
Difficulty: High Impact: Very High +15-20 pts
Overlay signal strength data on a Google Map in real-time. As the user walks or drives, the map fills in with color-coded coverage data (green = strong, yellow = medium, red = weak). Export heatmap as image. Server can aggregate heatmap data from ALL devices to build a crowd-sourced coverage map.
Why it impresses: Combines GPS + cellular APIs + Google Maps SDK. Visual and immediately useful for network operators. Similar to professional tools like OpenSignal and CellMapper.
Implementation Details
  • Add Google Maps SDK dependency + get API key from Google Cloud Console
  • Collect GPS coordinates (lat/lng) with each cell reading
  • Store coordinates in database alongside cell data
  • Render colored circles/tiles on map based on signal strength
  • Server endpoint to return all geo-tagged data for heatmap rendering
  • Requires: ACCESS_FINE_LOCATION (already needed), Google Maps API key
2. 5G (NR) Network Support
Difficulty: Medium Impact: Very High +10-15 pts
Extend the app beyond the 4G requirement to detect and report 5G NR cells. Report SS-RSRP, SS-RSRQ, SS-SINR, NR-ARFCN, and PCI. Show 5G as a separate category in all statistics.
Why it impresses: Goes beyond what's asked. Shows awareness of current and future network technologies. If you have a 5G-capable phone, you can collect real 5G data in Beirut.
Implementation Details
  • Check for CellInfoNr instances in getAllCellInfo()
  • Parse CellIdentityNr and CellSignalStrengthNr
  • Add "5G" as a network_type in the database and statistics
  • Requires: Android 10+ (API 29) and a 5G-capable phone
3. Machine Learning Signal Quality Prediction
Difficulty: High Impact: Very High +10-15 pts
Train a simple ML model on collected data to predict signal quality based on location, time of day, network type, and operator. Show predictions in the app: "Signal likely to drop in next 5 minutes."
Why it impresses: Combines telecom knowledge with data science. Publishable-quality feature for an undergrad project. Demonstrates practical application of ML to real-world network data.
Implementation Details
  • Collect sufficient data over several days of use
  • Use Python scikit-learn on server: Random Forest or Gradient Boosting
  • Features: lat, lng, hour_of_day, day_of_week, operator, network_type
  • Create GET /predict endpoint that returns quality forecast
  • Display prediction in app with confidence level
4. Network Speed Test Integration
Difficulty: Medium Impact: High +10-15 pts
Built-in download/upload speed test that correlates results with current cell parameters. Store: "On Alfa 4G Band 3 at -82 dBm, download speed was 45 Mbps." Show speed statistics alongside signal statistics.
Why it impresses: Makes the app immediately practical. Correlating speed with signal parameters is research-relevant. Professional apps like Speedtest by Ookla do exactly this.
Implementation Details
  • Download a known test file from server, measure throughput and time
  • Upload test data to server, measure upload speed
  • Record results alongside current cell parameters
  • Add speed charts (Mbps by network type, by operator, by signal level) to stats
5. Handover Detection & Analysis
Difficulty: Medium-High Impact: High +10-15 pts
Detect cell switches (handovers) and network type changes (e.g., 4G→3G fallback). Log every handover event with before/after cell info. Detect "ping-pong" handovers (rapid switching between 2 cells). Show handover frequency statistics.
Why it impresses: Directly relevant to mobile networks course material. Handover analysis is a professional network optimization task. Shows deep understanding of cellular network behavior.
Implementation Details
  • Compare consecutive CellInfo readings every 10 seconds
  • If cell_id or network_type changes, record a HandoverEvent
  • Model: old_cell, new_cell, old_type, new_type, timestamp, location
  • Dedicated handover statistics and timeline screen
6. Neighboring Cell Analysis
Difficulty: Medium Impact: High +5-10 pts
Display ALL detected cells (not just the serving cell). Show neighbor cells with signal strengths ranked strongest-first. Visualize which cells the phone might hand over to next.
Why it impresses: getAllCellInfo() returns ALL cells, not just the serving one. Demonstrates understanding of cellular topology. Useful for network planning.
Implementation Details
  • Filter getAllCellInfo() — isRegistered() separates serving from neighbors
  • Display neighbor list sorted by signal strength
  • Send neighbor data to server for aggregated cell mapping
7. Offline Mode with Local Database (Room)
Difficulty: Medium Impact: Medium-High +5-10 pts
Store data locally in Room/SQLite when no internet. Automatically sync to server when connection is restored. Show local statistics even offline. Never lose data.
Why it impresses: Professional-grade robustness. Room + LiveData demonstrates modern Android architecture. Most previous teams had no offline support.
Implementation Details
  • Add Room persistence library dependency
  • Define local DAO and Entity mirroring server schema
  • Implement sync: queue pending uploads, batch send on reconnect
  • Show sync status: "47 records pending upload"
8. Data Export & Sharing (CSV / PDF)
Difficulty: Low Impact: Medium +5 pts
Export collected cell data as a CSV file. Optionally generate a PDF report with charts. Share via email, WhatsApp, etc. using Android's share intent.
Implementation Details
  • Write data to CSV using StringBuilder
  • Use Android's FileProvider to share files securely
  • For PDF: use iText library or Android's built-in PDF APIs
9. Real-Time Server Push (WebSocket)
Difficulty: Medium Impact: Medium-High +5-10 pts
Bidirectional WebSocket communication. Server can push alerts to phones ("Signal dropped below threshold!"). Live dashboard updates without page refresh.
Implementation Details
  • Flask-SocketIO on server side
  • OkHttp WebSocket on Android side
  • Real-time event broadcasting for dashboard
10. Dual-SIM / Multi-SIM Support
Difficulty: Medium Impact: Medium +5-10 pts
Detect dual-SIM phones, report data from BOTH SIMs simultaneously, compare Alfa vs Touch side-by-side in real-time. Per-SIM statistics.
Implementation Details

11. Dark Mode + Material You Theming

Full dark mode support. Dynamic color theming (Material You on Android 12+). Smooth theme transitions. Huge visual polish for minimal effort.

Lessons from Previous Projects

Analysis of 9 previous team submissions from past semesters.

What Previous Teams Did Right

  • Used Java for Android + Python Flask for server (proven combo)
  • Flask + SQLite was the most common and simplest backend
  • Deployed on Render.com (free tier works)
  • Included: demo video, PPT, report, APK in submissions
  • Server code was typically 200-400 lines of Python

Common Mistakes to Avoid

  • Not testing 2G/3G (only testing 4G because it's default)
  • Hardcoding server URLs (make them configurable)
  • Not handling null/MAX_VALUE from CellInfo APIs (crash)
  • Not requesting permissions properly (instant crash)
  • Not having enough data collected before demo
  • Submitting the venv/ or .gradle/ folders (huge file size)
What Will Make You Stand Out:
• Implementing 3-5 innovative features from Section 11
• Having a polished Material Design UI (most teams had basic UIs)
• Charts for statistics instead of just text (MPAndroidChart)
• Working 5G support if you have a 5G phone
• GPS heatmap (no previous team had this)
• Clean, well-organized code with proper architecture
