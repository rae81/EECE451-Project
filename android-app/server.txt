Network Cell Analyzer Server Contract

This file is updated alongside the Android app so the backend can be implemented without guessing.

Current app scope
- Android app name: `NetworkCellAnalyzer`
- Package: `com.networkanalyzer.app`
- Default base URL: configurable in app settings, stored in `PREF_SERVER_URL`
- Auth model: bearer token stored locally; requests also include `device_id`
- Collection cadence: every 10 seconds by default
- Offline mode: readings are stored in Room first and retried later by `OfflineSyncWorker`
- Foreground monitor: `CellMonitorService` reads the currently registered serving cell, stores it locally, broadcasts it to the dashboard UI, and tries to upload immediately
- Guest mode: app can be entered without server authentication via an offline path from the login screen
- Local stats fallback: if server stats are unavailable, the app computes operator percentages, network percentages, average signal, and average SNR from the local Room database
- Radio-quality field note: Android still sends the JSON key as `snr`, but the value is really the best available radio quality metric for the current RAT. Today that means LTE RSSNR/SINR, NR SINR, WCDMA Ec/No when the device exposes it, and `-999` when the platform provides no usable quality metric
- Speed test fallback: if the app is still using the placeholder base URL, mobile runs latency/download/upload tests against public internet endpoints and does not try to POST the result to the backend. Current fallback endpoints are public HTTPS endpoints chosen only so the mobile app can function without your server during development
- Tower-database prep: Android now stores and uploads `mcc`, `mnc`, `lac`, `cell_id`, and `frequency_band` together so a future backend or external tower database can estimate serving-cell locations. The in-app map still shows measurement locations, not true tower coordinates

Endpoints currently expected by the app

1. `POST /auth/register`
- Request JSON:
```json
{
  "name": "User Name",
  "email": "user@example.com",
  "password": "secret123",
  "device_id": "uuid"
}
```
- Success response JSON:
```json
{
  "success": true,
  "token": "jwt-access-token",
  "refresh_token": "jwt-refresh-token",
  "user_name": "User Name",
  "user_email": "user@example.com",
  "message": "registered"
}
```

2. `POST /auth/login`
- Request JSON:
```json
{
  "email": "user@example.com",
  "password": "secret123",
  "device_id": "uuid"
}
```
- Success response JSON: same shape as register response

3. `POST /auth/refresh`
- Request JSON:
```json
{
  "refresh_token": "jwt-refresh-token"
}
```
- Response JSON: same auth response shape

4. `POST /receive-data`
- Sent by the foreground monitor when online
- Request JSON:
```json
{
  "device_id": "uuid",
  "operator": "Touch",
  "signal_power": -88,
  "snr": 11.2,
  "network_type": "4G",
  "frequency_band": "1800",
  "cell_id": "12345",
  "lac": "42",
  "mcc": "415",
  "mnc": "03",
  "latitude": 33.8938,
  "longitude": 35.5018,
  "timestamp": "2026-03-10T14:12:00.000Z",
  "ip_address": "optional",
  "mac_address": "optional",
  "sim_slot": 0
}
```
- Success response JSON:
```json
{
  "success": true,
  "message": "stored"
}
```

5. `POST /receive-data/batch`
- Sent by offline sync worker when connectivity returns
- Request JSON:
```json
{
  "device_id": "uuid",
  "data": [
    {
      "device_id": "uuid",
      "operator": "Touch",
      "signal_power": -88,
      "snr": 11.2,
      "network_type": "4G",
      "frequency_band": "1800",
      "cell_id": "12345",
      "lac": "42",
      "mcc": "415",
      "mnc": "03",
      "latitude": 33.8938,
      "longitude": 35.5018,
      "timestamp": "2026-03-10T14:12:00.000Z",
      "sim_slot": 0
    }
  ]
}
```
- Response JSON:
```json
{
  "success": true,
  "message": "batch stored"
}
```

6. `GET /get-stats`
- Query params:
  - `device_id`
  - `from` as epoch milliseconds string
  - `to` as epoch milliseconds string
- Response JSON:
```json
{
  "success": true,
  "operator_time": {
    "Touch": 62.0,
    "Alfa": 38.0
  },
  "network_type_time": {
    "4G": 70.0,
    "3G": 20.0,
    "2G": 10.0
  },
  "avg_signal_per_type": {
    "4G": -86.5,
    "3G": -91.2
  },
  "avg_signal_per_device": {
    "uuid": -88.0
  },
  "avg_snr_per_type": {
    "4G": 12.8,
    "3G": 7.1
  },
  "total_records": 1542,
  "from_date": "2026-03-01",
  "to_date": "2026-03-10"
}
```

7. `GET /get-stats/avg-all`
- Query params: `from`, `to`
- Response shape: same as `/get-stats`, but aggregated across all devices

8. `GET /central-stats`
- Used for central dashboard
- Can return either JSON or HTML, but JSON support is preferred if mobile uses it later

9. `POST /speed-test`
- Request JSON:
```json
{
  "downloadMbps": 45.2,
  "uploadMbps": 13.7,
  "latencyMs": 32,
  "timestamp": 1773151800000
}
```
- Current Android implementation sends only the four fields above
- Mobile only calls this endpoint when a real backend base URL is configured; in guest/placeholder mode it keeps speed-test results local and uses public endpoints for the actual measurements
- Recommended server extension: also accept `device_id`, current operator, network type, and signal data for richer analytics without breaking the current payload

10. `GET /device/profile`
- Query params: `device_id`
- Response should include at minimum device metadata and summary stats

Auth expectations
- Android sends bearer tokens via interceptor when a token exists
- Backend should tolerate unauthenticated local development only if explicitly configured
- Production recommendation: secure all non-demo endpoints except maybe health checks
- Mobile behavior note: the app is no longer blocked by missing auth backend because it supports offline guest entry

Database notes
- Store each reading with:
  - `device_id`
  - `operator`
  - `signal_power`
  - `snr`
  - `network_type`
  - `frequency_band`
  - `cell_id`
  - `lac`
  - `mcc`
  - `mnc`
  - `latitude`
  - `longitude`
  - `timestamp`
  - `sim_slot`
- Add indices on `device_id`, `timestamp`, `network_type`, `operator`
- Backend interpretation note: do not hard-code `snr` as LTE-only SNR. Treat it as a numeric quality metric in dB whose exact radio meaning depends on `network_type`

Implementation notes for backend teammate
- Keep the base URL stable and configurable; Android reads it from settings
- Return `application/json` for API endpoints
- Use ISO-8601 timestamps on write endpoints where possible
- Do not rename JSON fields unless Android models are updated too
- Mobile map behavior note: current map popups group readings by collection location and show the serving cell IDs, PLMN (`mcc`-`mnc`), and channels seen there. True tower positions still require a backend-side lookup against an external cell database
